# Introduction

High-quality documentation is essential for our project. This guide explains how to contribute effectively using our structured, topic-based approach.

Our core principle is that the /docs directories should contain only MDX files.

## Documentation Principles

**Documentation reflects current functionality only.** All documentation must accurately describe the current state of the CMS. Do not document planned features, future improvements, or changes that haven't been implemented yet.

- ‚úÖ **Document what exists now**: Features, functions, and behaviors that are currently in the codebase
- ‚ùå **Do not document future plans**: Improvements, changes, or features go in dedicated change logs or issue trackers
- üîÑ **Keep docs synchronized**: Update documentation immediately when functionality changes

This ensures users always have accurate, reliable information about what the CMS can do right now.

Documentation Structure: A "Book" Analogy
Think of our documentation as a digital book, organized for clarity and easy maintenance. The structure is based directly on the file system, which means the folder hierarchy you see in the code is exactly what you'll see in the navigation.

The Library: /docs
This is the root directory for all documentation content.

The Bookshelves: Top-Level Categories
We have three main "bookshelves," each for a specific audience:

/docs/user-guide/ (The User Manual): This is for people using the CMS. It covers topics like creating content, managing media, and configuring site settings.

/docs/dev-guide/ (The Developer Handbook): This is for people building with or extending the CMS. It includes installation guides, API references, and theming tutorials.

/docs/contributing/ (The Contributor's Guide): This is for people who want to improve the project. It contains our code of conduct, pull request guidelines, and this very document.

The Chapters and Pages
Within each bookshelf, content is grouped into "chapters" (topic folders). Every folder must have an index.mdx file.

index.mdx: This is the title page or introduction for that section. It provides a brief overview of the content within that folder.

Content Files (\*.mdx): These are the individual articles or "pages" within a chapter, like rich-text-editor.mdx.

A Complete Example
Here‚Äôs how these pieces fit together to create a clear, navigable structure:

docs/
‚îî‚îÄ‚îÄ user-guide/ # The "User Manual" bookshelf
‚îú‚îÄ‚îÄ index.mdx # Landing page for the User Guide
‚îî‚îÄ‚îÄ content-editing/ # A "chapter" on content editing

‚îú‚îÄ‚îÄ rich-text-editor.mdx # Page 1 in this chapter
‚îî‚îÄ‚îÄ managing-images.mdx # Page 2 in this chapter

This "book" approach ensures that anyone can find the information they need and contributors know exactly where to place new content.

Frontmatter Guide: The Page's DNA
Every MDX file must begin with a YAML frontmatter block, enclosed by --- lines. This block is the "brain" of the page‚Äîit provides the critical metadata our CMS needs to build the navigation, display information correctly, and enable features like search and filtering.

Field Breakdown
Here is a detailed breakdown of each required field. Please maintain this exact order for consistency.

| Field       | Required | Purpose                                                                                             | Example Value                       |
| ----------- | -------- | --------------------------------------------------------------------------------------------------- | ----------------------------------- |
| path        | Yes      | The full, unique path to the file from the project's root. Crucial for linking and identification.  | "docs/dev-guide/api/endpoints.mdx"  |
| title       | Yes      | The main heading (H1) of the page. Displayed in the navigation, browser tab, and search results.    | "API Endpoints"                     |
| description | Yes      | A concise, one-sentence summary. Used for SEO and as a preview in search results and link cards.    | "A detailed guide to the REST API." |
| order       | Yes      | A number that determines the page's position in the sidebar navigation, relative to its siblings.   | 2                                   |
| icon        | Yes      | The Material Design Icon to display next to the title in the navigation. Use the mdi: prefix.       | "mdi:api"                           |
| author      | Yes      | The GitHub username of the original author.                                                         | "YourGitHubUsername"                |
| created     | Yes      | The date the document was first created, in YYYY-MM-DD format.                                      | "2025-09-28"                        |
| updated     | Yes      | The date the document was last significantly modified. Update this with every meaningful change.    | "2025-10-01"                        |
| tags        | Yes      | A list of relevant keywords to aid in search and filtering. Use lowercase and kebab-case if needed. | - "api" - "developer" - "endpoints" |

Complete Template
Use this as a starting point for any new documentation file. Just copy, paste, and fill in the values.

```yaml
---
path: 'docs/section/sub-section/your-file-name.mdx'
title: 'A Clear and Descriptive Title'
description: "A one-sentence summary of the document's purpose."
order: 3
icon: 'mdi:icon-name'
author: 'YourGitHubUsername'
created: 'YYYY-MM-DD'
updated: 'YYYY-MM-DD'
tags:
  - 'relevant-tag'
  - 'another-tag'
---
```

Managing Assets (Images, GIFs, etc.)
To maintain a clean and organized repository, all non-MDX files (such as images, diagrams, or videos) must be placed in a central static directory.

Asset Location: /static/docs/

Structure: Mirror the /docs directory structure for your assets. For example, an image used in /docs/user-guide/content-editing/ should be placed in /static/docs/user-guide/content-editing/.

When you need to use an image in your document, reference it using its absolute path from the root.

Correct Usage:

```javascript
{/* Place the image 'managing-images.png' in '/static/docs/user-guide/content-editing/' */}
![How to manage images](/docs/user-guide/content-editing/managing-images.png)
```

This policy ensures that our documentation folders contain only .mdx source files, making the repository significantly easier to navigate and maintain.

Content Best Practices
To ensure our documentation is clear, helpful, and professional, follow these guidelines when writing:

- **Clarity and Simplicity**: Use active voice, short sentences, and plain language. Avoid jargon; explain terms if needed.
- **Structure**: Start with an introduction, use headings (H2, H3) for sections, and end with a summary or next steps.
- **Examples**: Provide practical examples, code snippets, and screenshots where helpful.
- **Consistency**: Follow the same tone and style as existing docs. Use American English.
- **Accessibility**: Add alt text to images, ensure good contrast, and make content scannable.

MDX-Specific Guidance
Since our docs use MDX (Markdown with JSX), you can enhance content with components, but keep it minimal:

- **Basic Markdown**: Use standard syntax for bold, italics, lists, links, and code blocks (e.g., ```javascript for syntax highlighting).
- **JSX Components**: If needed, import and use Svelte components, but only for interactive elements‚Äîmost docs should stick to Markdown.
- **Escaping**: For literal angle brackets in code, use backticks or HTML entities (&lt; &gt;).
- **Imports**: At the top of the file, after frontmatter, you can import components if required.

Expanded Image and Media Guidelines
Building on the asset policy:

- **Formats**: Use PNG for screenshots, SVG for icons/diagrams, and optimize for web (e.g., under 500KB).
- **Naming**: Use descriptive, kebab-case names (e.g., `api-endpoints-diagram.svg`).
- **Alt Text**: Always provide meaningful alt text for accessibility.
- **Placement**: Mirror /docs/ structure in /static/docs/. Reference with absolute paths.
- **Optimization**: Compress images before committing to keep the repo size down.

Review and Update Process

- **PR Reviews**: All doc changes go through PRs. Reviewers check for accuracy, clarity, and adherence to guidelines.
- **Updating Metadata**: Always update the `updated` field in frontmatter on changes.
- **Versioning**: Docs are versioned with code; major changes may need release notes.
- **Feedback**: Use issues or PR comments for suggestions.

# Installation Guide

Follow these steps to install SveltyCMS.

## Prerequisites

- Node.js 18+
- Bun package manager

## Steps

1. Clone the repo: `git clone https://github.com/SveltyCMS/SveltyCMS.git`
2. Install dependencies: `bun install`
3. Run the dev server: `bun run dev`

For more help, see the [troubleshooting section](#troubleshooting).

## Troubleshooting

If you encounter issues, check the logs or open an issue.

Before submitting a PR for new documentation:

- [ ] Tested rendering in the CMS.
- [ ] `updated` field is current.

Introduction
High-quality documentation is essential for our project. This guide explains how to contribute effectively using our structured, topic-based approach. Our core principle is that the /docs directories should contain only MDX files.
Documentation Structure: A "Book" Analogy
Think of our documentation as a digital book, organized for clarity and easy maintenance. The structure is based directly on the file system, which means the folder hierarchy you see in the code is exactly what you'll see in the navigation
The Library: /docs
This is the root directory for all documentation content.
The Bookshelves: Top-Level Categories
We have three main "bookshelves," each for a specific audience:

- /docs/user-guide/ (The User Manual): This is for people using the CMS. It covers topics like creating content, managing media, and configuring site settings.
- /docs/dev-guide/ (The Developer Handbook): This is for people building with or extending the CMS. It includes installation guides, API references, and theming tutorials.
- /docs/contributing/ (The Contributor's Guide): This is for people who want to improve the project. It contains our code of conduct, pull request guidelines, and this very document.
  The Chapters and Pages
  Within each bookshelf, content is grouped into "chapters" (topic folders). Every folder must have an index.mdx file.
- index.mdx: This is the title page or introduction for that section. It provides a brief overview of the content within that folder.
- Content Files (\*.mdx): These are the individual articles or "pages" within a chapter, like rich-text-editor.mdx.

  docs/

  Content Best Practices
  To ensure our documentation is clear, helpful, and professional, follow these guidelines when writing:

- **Clarity and Simplicity**: Use active voice, short sentences, and plain language. Avoid jargon; explain terms if needed.
- **Structure**: Start with an introduction, use headings (H2, H3) for sections, and end with a summary or next steps.
- **Examples**: Provide practical examples, code snippets, and screenshots where helpful.
- **Consistency**: Follow the same tone and style as existing docs. Use American English.
- **Accessibility**: Add alt text to images, ensure good contrast, and make content scannable.

MDX-Specific Guidance
Since our docs use MDX (Markdown with JSX), you can enhance content with components, but keep it minimal:

- **Basic Markdown**: Use standard syntax for bold, italics, lists, links, and code blocks (e.g., ```javascript for syntax highlighting).
- **JSX Components**: If needed, import and use Svelte components, but only for interactive elements‚Äîmost docs should stick to Markdown.
- **Escaping**: For literal angle brackets in code, use backticks or HTML entities (&lt; &gt;).
- **Imports**: At the top of the file, after frontmatter, you can import components if required.

Expanded Image and Media Guidelines
Building on the asset policy:

- **Formats**: Use PNG for screenshots, SVG for icons/diagrams, and optimize for web (e.g., under 500KB).
- **Naming**: Use descriptive, kebab-case names (e.g., `api-endpoints-diagram.svg`).
- **Alt Text**: Always provide meaningful alt text for accessibility.
- **Placement**: Mirror /docs/ structure in /static/docs/. Reference with absolute paths.
- **Optimization**: Compress images before committing to keep the repo size down.

Review and Update Process

- **PR Reviews**: All doc changes go through PRs. Reviewers check for accuracy, clarity, and adherence to guidelines.
- **Updating Metadata**: Always update the `updated` field in frontmatter on changes.
- **Versioning**: Docs are versioned with code; major changes may need release notes.
- **Feedback**: Use issues or PR comments for suggestions.
