### Store Pattern Example

```typescript
// stores/exampleStore.svelte.ts
import { writable, derived } from 'svelte/store';

// Reactive state
export const items = writable<Item[]>([]);
export const loading = writable(false);

// Derived state
export const itemCount = derived(items, ($items) => $items.length);

// Actions
export const actions = {
	async loadItems() {
		loading.set(true);
		try {
			const data = await api.getItems();
			items.set(data);
		} finally {
			loading.set(false);
		}
	}
};
```

## 📝 File Naming Conventions

### SvelteKit Conventions

- `+page.svelte` - Page components
- `+layout.svelte` - Layout components
- `+page.server.ts` - Server-side page logic
- `+layout.server.ts` - Server-side layout logic
- `+server.ts` - API endpoints
- `+error.svelte` - Error pages

### Component Conventions

- `ComponentName.svelte` - PascalCase for components
- `componentName.ts` - camelCase for utilities
- `component-name.css` - kebab-case for styles
- `ComponentName.test.ts` - Test files
- `ComponentName.stories.ts` - Storybook stories

### Directory Conventions

- `kebab-case/` - URL-safe directory names
- `camelCase/` - JavaScript-style names
- `PascalCase/` - Component directories

## 🔗 Import Patterns

### Path Aliases

```typescript
// Configured in vite.config.ts and tsconfig.json
import Widget from '@widgets/Button/Button.svelte';
import { authStore } from '@stores/authStore.svelte';
import { validateInput } from '@utils/validation';
import type { User } from '@types/user';
```

### Import Organization

```typescript
// 1. External dependencies
import { writable } from 'svelte/store';
import { error } from '@sveltejs/kit';

// 2. Internal utilities
import { db } from '@databases';
import { logger } from '@utils/logger';

// 3. Types
import type { Widget, WidgetConfig } from '@types';

// 4. Components (in .svelte files)
import Button from '@widgets/Button/Button.svelte';
```

## 🔄 Development Workflow

### File Organization Best Practices

1. **Feature-based grouping** - Related files stay together
2. **Consistent naming** - Follow established conventions
3. **Clear separation** - Client vs server, components vs utilities
4. **Logical hierarchy** - Nested structure reflects relationships
5. **Documentation** - README files for complex directories

### Adding New Features

1. Create feature directory in appropriate location
2. Follow naming conventions
3. Add necessary configuration
4. Write tests
5. Update documentation
6. Export from index files

This structure supports scalable development while maintaining clear separation of concerns and following SvelteKit best practices.
